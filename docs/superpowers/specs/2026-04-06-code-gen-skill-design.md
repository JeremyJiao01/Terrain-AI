# Code-Gen Skill 设计文档

> 设计一个 Skill，引导外部 Agent（MCP client）在收到完善的设计文档后，通过 MCP 工具对目标代码库进行深度调研，输出结构化的实施方案。

## 设计约束

| 项目 | 决策 |
|------|------|
| 输入 | 已完善的设计文档（经 brainstorming 输出的结构化 spec） |
| 载体 | Skill 文件（Agent 侧加载） |
| 输出 | 结构化实施方案（停下等用户确认，不写代码） |
| 调研深度 | 深度（含上下游调用链分析） |
| 控制方式 | 固定骨架 + 局部自主（4 阶段 + 1 补漏回路） |

## 整体架构

```
┌─────────────────────────────────────────────────┐
│  阶段 1：概念提取                                  │
│  输入: 设计文档                                    │
│  输出: 关键概念列表 (2-8 个关键词/短语)             │
│  工具: 无 (纯 Agent 推理)                          │
├─────────────────────────────────────────────────┤
│  阶段 2：广度搜索                                  │
│  输入: 关键概念列表                                │
│  输出: 候选接口列表 (qualified_name + 初步相关性)   │
│  工具: find_api × N (每个概念搜一次)               │
├─────────────────────────────────────────────────┤
│  阶段 3：深度调研                                  │
│  输入: 候选接口列表                                │
│  输出: 接口画像 (签名、调用树、调用者、代码风格)    │
│  工具: get_api_doc / find_callers / trace_call_chain │
│        list_api_docs (确认模块层级)                 │
├───────────────┬─────────────────────────────────┤
│  阶段 3.5：补漏检查                                │
│  如果调研中发现设计文档涉及但阶段2未覆盖的接口       │
│  → 回到阶段 2 补搜 (最多 1 轮)                     │
│  否则 → 进入阶段 4                                 │
├─────────────────────────────────────────────────┤
│  阶段 4：方案输出                                  │
│  输入: 全部调研结果                                │
│  输出: 结构化实施方案                              │
│  工具: 无 (纯 Agent 综合)                          │
│  ⚠️ STOP — 等待用户确认                           │
└─────────────────────────────────────────────────┘
```

### 阶段间数据流

```
设计文档 ──→ [阶段1] ──concepts──→ [阶段2] ──candidates──→ [阶段3]
                                                              │
                                        ┌──new_concepts───[阶段3.5]
                                        │                     │
                                        ↓                     ↓ (无遗漏)
                                    [阶段2补搜]→[阶段3补调研]→[阶段4] ──→ 实施方案
```

### 控制规则

- 阶段顺序不可跳过
- 补漏回路最多执行 1 轮
- 阶段 4 输出后必须停下等待用户确认
- 每个阶段内部，Agent 自主决定工具调用的参数和次数

## 阶段详细设计

### 阶段 0：环境检查

Skill 启动后，先调用 `get_repository_info` 验证 MCP 中有活跃仓库且 graph、api_docs、embeddings 均可用。

如果缺失，停下提示：
> "Repository index is incomplete. Please run `/repo-init <repo-path>` first."

### 阶段 1：概念提取

Agent 阅读设计文档，提取：

- **功能概念**：功能关键词（如"定时保存"、"串口通信"、"故障检测"）
- **实体名称**：提到的具体模块名、函数名、类型名
- **行为动词**：初始化、注册、回调、轮询等——暗示需要搜索的接口类型

输出格式（Agent 内部使用，不呈现给用户）：

```yaml
concepts:
  - keyword: "串口初始化"
    reason: "设计文档要求新模块通过串口发送数据"
  - keyword: "故障码注册"
    reason: "需要将新故障类型注册到现有故障管理系统"
```

**局部自主**：概念数量由 Agent 根据设计文档复杂度判断（2-8 个）。

### 阶段 2：广度搜索

对每个概念调用 `find_api`：

```
find_api(query="串口初始化", top_k=5)
find_api(query="故障码注册", top_k=5)
```

Agent 从返回结果中筛选候选列表：
- 保留相关性高的（语义得分 + Agent 判断）
- 去重（同一 qualified_name 可能被多个搜索命中）
- 标记每个候选与设计文档的关联点

**局部自主**：`top_k` 可调；某个概念搜不到结果时可换关键词重试一次。

### 阶段 3：深度调研

对候选列表中的每个接口，按需调用：

| 目的 | 工具 | 何时调用 |
|------|------|---------|
| 获取完整签名、调用树、源码 | `get_api_doc` | 每个候选必调 |
| 查谁调用了这个接口 | `find_callers` | 需要理解使用模式时 |
| 追溯完整调用链 | `trace_call_chain` | 需要确认入口点或影响范围时 |
| 确认模块层级结构 | `list_api_docs` | 至少调一次，确认新函数放置位置 |

从调研结果中提取：

1. **可复用接口**：确认签名、参数含义、调用前置条件
2. **调用模式**：从 `find_callers` 结果看其他代码如何调用该接口（参数传递、错误处理）
3. **代码风格**：从 `get_api_doc` 返回的源码归纳命名、注释、错误处理规范
4. **依赖方向**：从调用树确认新代码调用老接口是合法的（不逆向依赖）

**局部自主**：哪些候选需要 `trace_call_chain`、哪些只需 `find_callers`，由 Agent 判断。

### 阶段 3.5：补漏检查

Agent 回顾阶段 3 调研结果，检查：

- `get_api_doc` 返回的调用树中，是否有新出现的关键接口未在阶段 2 搜索过？
- `find_callers` 返回的调用者中，是否暗示了设计文档遗漏的依赖？

**遗漏判定标准**：调研结果中出现的 qualified_name，如果满足以下两个条件之一，视为遗漏：
1. 该接口在设计文档中被直接或间接提及，但阶段 2 未搜索到
2. 该接口是候选接口的直接依赖（callees），且新代码可能需要直接调用它

如果发现遗漏 → 对新发现的接口回到阶段 2 补搜 + 阶段 3 补调研（**仅 1 轮**）

如果无遗漏 → 直接进入阶段 4

### 阶段 4：方案输出

综合所有调研结果，输出固定格式的实施方案：

```markdown
# 实施方案

## 实现目标
[一段话总结，来自设计文档]

## 复用的已有接口
| 接口 | 签名 | 文件位置 | 调用方式说明 |
|------|------|---------|-------------|
| `qualified_name` | `signature` | `path:line` | 如何调用、注意事项 |

## 需要新增的函数
| 函数名 | 所属模块/文件 | 职责 | 依赖的已有接口 |
|--------|-------------|------|---------------|
| `new_func` | `path` | 做什么 | 调用哪些 |

## 需要修改的文件
| 文件 | 修改内容 | 原因 |
|------|---------|------|
| `path` | 改什么 | 为什么 |

## 依赖顺序
file_a → file_b → file_c

## 代码风格约定
- 命名规范: ...
- 错误处理: ...
- 注释语言: ...

## 架构约束
- 依赖方向: ...
- 层级归属: ...
```

**⚠️ STOP — 呈现给用户，等待确认后才可进行下一步。**

## 使用的 MCP 工具清单

| 工具 | 阶段 | 用途 |
|------|------|------|
| `get_repository_info` | 0 | 验证环境就绪 |
| `find_api` | 2, 3.5 | 语义搜索候选接口 |
| `get_api_doc` | 3 | 获取接口详细文档（签名、调用树、源码） |
| `find_callers` | 3 | 查找接口的调用方 |
| `trace_call_chain` | 3 | 追溯完整调用链 |
| `list_api_docs` | 3 | 确认模块层级结构 |

不需要新增或修改任何 MCP 工具，现有暴露的工具已覆盖全部需求。

## 边界情况

- **`find_api` 无结果**：换关键词重试一次；仍无结果则标记该概念为"无现有实现"，在方案中注明需从零实现。
- **设计文档涉及索引外的语言**：停下告知用户，建议先索引目标仓库。
- **设计文档过大**：建议拆分为多次 `/code-gen` 调用，每次聚焦一个子功能。
- **候选接口过多（>15个）**：Agent 按相关性排序，优先调研前 10 个，其余在补漏阶段按需补充。
