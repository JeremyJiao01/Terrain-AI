# /code-gen Skill 设计文档

## 概述

`/code-gen` 是一个独立的 Claude Code Skill（slash command），接收自由文本需求描述或需求文档路径，指导 agent 利用 CodeGraphWiki 的 MCP 工具收集项目上下文，然后生成跨文件的、与项目风格/接口/架构完全对齐的代码。

### 核心决策

- **Skill 而非 MCP tool** — 代码生成由外层 agent（Claude Code）完成，CodeGraphWiki 的 MCP 工具负责"理解代码"，agent 负责"写代码"。不在 MCP 内部调用 LLM 生成代码，避免 LLM 套 LLM 的成本浪费。
- **完全独立** — 不引用、不依赖任何其他 Skill（如 `/api-find`、`/cgb-start`），直接通过 `cgb_cli.py` 子命令与底层工具交互。
- **跨文件粒度** — 支持生成涉及多个新建文件 + 已有文件修改的完整 feature。
- **全面对齐** — 风格（命名、缩进、注释）+ 接口（签名、参数类型、导入路径）+ 架构（分层模式、目录位置、依赖方向）。

### 触发方式

```
/code-gen <需求文档路径或自由文本需求描述>
```

### 前置条件

仓库已通过 `/repo-init` 或 `initialize_repository` MCP tool 完成索引（图谱 + API 文档 + 嵌入向量就绪）。Skill 开头通过 `cgb_cli.py info` 检查状态，未就绪则终止并提示用户。

## 工作流程

### 阶段 1：环境检查

```bash
python3 ~/.claude/commands/code-graph/cgb_cli.py info
```

- 确认有活跃仓库且索引完整（图谱、API 文档、嵌入向量）
- 如果未就绪，终止并告知用户需要先初始化仓库

### 阶段 2：需求理解

- 如果 `$ARGUMENTS` 是一个存在的文件路径 → 读取文件内容作为需求
- 如果是自由文本 → 直接使用
- Agent 从需求中提取：
  - **功能目标** — 要实现什么
  - **涉及的概念/实体** — 关键词（如"认证"、"缓存"、"路由"）
  - **预期的输入输出** — 接口行为

### 阶段 3：上下文收集

Agent 使用以下命令多次查询项目知识图谱：

1. **语义搜索** — 用提取的关键概念搜索相关实现
   ```bash
   python3 ~/.claude/commands/code-graph/cgb_cli.py api-find "<关键词>"
   ```

2. **接口详情** — 对高相关度的结果获取完整接口信息
   ```bash
   python3 ~/.claude/commands/code-graph/cgb_cli.py api-detail "<qualified_name>"
   ```

3. **模块结构** — 了解项目的模块组织和分层模式
   ```bash
   python3 ~/.claude/commands/code-graph/cgb_cli.py api-docs
   ```

4. **依赖关系（按需）** — 查询图谱了解依赖方向
   ```bash
   python3 ~/.claude/commands/code-graph/cgb_cli.py query "<cypher查询>"
   ```

**搜索策略**：
- 从需求中提取 2-5 个关键概念，分别搜索
- 对每个搜索结果中 score 最高的 1-2 个函数获取详情
- 通过模块结构识别项目分层模式和目录规范

### 阶段 4：变更规划

Agent 基于收集到的上下文，输出一份变更计划：

```
## 变更计划

### 新建文件
1. `<路径>` — <职责说明>
2. `<路径>` — <职责说明>

### 修改文件
1. `<路径>` — <修改内容概述>
2. `<路径>` — <修改内容概述>

### 依赖顺序
<文件A> → <文件B> → <文件C>
```

**用户确认门控**：输出计划后必须等待用户确认，用户可以要求修改计划。只有用户确认后才进入阶段 5。

### 阶段 5：逐文件生成

- 按依赖顺序逐个生成代码
- 每个文件生成时，agent 的上下文中包含：
  - 已有接口的准确签名（来自阶段 3 的 api-detail 结果）
  - 前序已生成文件的接口定义
  - 项目命名规范样本（来自搜索到的代码）
- 生成完毕后展示完整变更清单供用户审查

## 对齐约束

### 风格对齐
- 从搜索到的代码样本中提取命名规范（snake_case / camelCase / etc.）
- 匹配项目的导入风格（相对/绝对导入、导入顺序）
- 匹配注释语言和文档字符串风格

### 接口对齐
- 调用已有函数时，必须使用从 `api-detail` 获取的准确签名
- 不得猜测参数名或类型，必须基于工具返回的实际接口
- 导入路径基于项目实际的模块结构

### 架构对齐
- 从模块结构中识别项目分层（如 service → tool → handler）
- 新文件放置在符合已有目录结构的位置
- 依赖方向与已有代码一致（不引入反向依赖）

## 边界情况

| 场景 | 行为 |
|------|------|
| 仓库未初始化 | 终止，提示用户先执行初始化 |
| `api-find` 无匹配结果 | 扩大搜索关键词重试一次，仍无结果则告知用户该项目中无相关代码，基于需求从零生成 |
| 需求涉及项目不支持的语言 | 告知用户，按需求语言的通用最佳实践生成（不保证架构对齐） |
| 需求文档过大 | 建议用户拆分为多次 `/code-gen` 调用 |
| 用户在阶段 4 拒绝计划 | 根据用户反馈修改计划，重新提交确认 |

## 实现方式

Skill 文件：`.claude/commands/code-gen.md`

文件内容为纯 Markdown prompt，包含：
1. 角色定义
2. 环境检查的 bash 命令
3. 输入解析规则（判断 $ARGUMENTS 是路径还是文本）
4. 上下文收集的命令和搜索策略
5. 变更规划的输出模板
6. 对齐约束的具体规则
7. 用户确认门控指令
8. 逐文件生成的指导

不涉及任何新的 Python 代码或 MCP tool 变更。
